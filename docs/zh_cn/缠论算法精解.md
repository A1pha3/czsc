## 一、CZSC 分析器架构

### 1.1 核心数据流

`CZSC` 类是整个缠论分析的入口。它接收原始 K 线，输出分型、笔、中枢等分析结果。

```
输入：List[RawBar]
                │
                ▼
        ┌───────────────┐
        │  CZSC.__init__ │
        │                │
        │  bars_raw ─────┼──→ 原始 K 线存储
        │  bars_ubi ─────┼──→ 未完成笔的无包含 K 线
        │  bi_list ──────┼──→ 已确认的笔列表
        │  cache ────────┼──→ 信号计算缓存
        │                │
        └───────┬────────┘
                │
                ▼
        ┌───────────────┐
        │  CZSC.update() │ ← 核心方法，每根新 K 线调用一次
        │                │
        │  1. 更新 bars_raw
        │  2. 去除包含关系 → bars_ubi
        │  3. __update_bi → bi_list
        │  4. 数量控制（内存管理）
        │  5. 计算信号（如有回调）
        │                │
        └────────────────┘
```

### 1.2 核心属性详解

```python
class CZSC:
    """缠中说禅分析器"""
    
    # ===== 输入数据 =====
    bars_raw: List[RawBar]      # 原始 K 线序列（只追加，不修改已有的）
    
    # ===== 中间状态 =====
    bars_ubi: List[NewBar]      # 未完成笔的无包含 K 线序列
                                # "ubi" = Unfinished BI
                                # 这是一个"工作区"——新 K 线先在这里处理，
                                # 直到形成一根完整的笔后才"转移"到 bi_list

    # ===== 分析结果 =====
    bi_list: List[BI]           # 已确认的笔列表
    
    # ===== 配置 =====
    max_bi_num: int = 50        # 最大保留笔数（内存控制）
    
    # ===== 信号 =====
    cache: OrderedDict          # 信号计算缓存
    get_signals: Callable       # 信号计算回调函数
```

> 💡 **关键理解**：`bars_ubi` 是"未成形的笔"——就像陶艺师手中正在塑形的泥巴。一旦满足成笔条件，就会"烧制定型"放入 `bi_list`。而 `bi_list` 中的笔也不是绝对不可变的——如果后续走势证明判断有误（笔破坏），最后一笔会被"打碎"回到 `bars_ubi` 重新塑形。

#### 状态机视角

从状态机的角度理解 CZSC 更直观：

```
                    ┌──────────────────────────────┐
                    │                              │
                    ▼                              │
    ┌──────────────────────┐  成笔    ┌───────────┴──────────┐
    │  状态A: 寻找笔       │ ────→   │  状态B: 笔已确认     │
    │  (bars_ubi 在积累)   │         │  (bi_list 有新笔)    │
    │                      │ ←────   │                      │
    └──────────────────────┘  笔破坏  └──────────────────────┘
                    │                              │
              来新K线│                        来新K线│
                    ▼                              ▼
              去除包含关系                    去除包含关系
              检查是否成笔                    检查是否被破坏
```

## 二、update() 方法：逐行解析

### 2.1 Step 1：更新原始 K 线

```python
def update(self, bar: RawBar):
    # Step 1: 更新 bars_raw
    if not self.bars_raw or bar.dt != self.bars_raw[-1].dt:
        # 新时间 → 追加
        self.bars_raw.append(bar)
        last_bars = [bar]
    else:
        # 同时间 → 替换最后一根
        # 场景：实时行情中，同一根 K 线在时间段内会不断更新
        self.bars_raw[-1] = bar
        last_bars = self._rebuild_last()  # 重建最后几根 K 线的状态
```

> **为什么支持"同时间替换"？** 在实盘场景中，一根 30 分钟 K 线在 10:00-10:30 期间会不断更新（每笔成交都会改变 Close、High、Low、Vol）。CZSC 需要能正确处理这种"K 线延伸"。

### 2.2 Step 2：去除包含关系

```python
    # Step 2: 对 last_bars 中的每根 bar 去除包含关系
    for bar in last_bars:
        if len(self.bars_ubi) < 2:
            # 不足 2 根，直接加入
            self.bars_ubi.append(NewBar(bar))
        else:
            # 取最后两根无包含 K 线
            k1, k2 = self.bars_ubi[-2], self.bars_ubi[-1]
            has_include, k4 = remove_include(k1, k2, bar)
            
            if has_include:
                # 有包含 → 合并后替换 k2
                self.bars_ubi[-1] = k4
            else:
                # 无包含 → 作为新的独立 K 线加入
                self.bars_ubi.append(k4)
```

#### 可视化流程

```
bars_ubi = [..., k1, k2]，新来 bar：

  情况A：有包含
    [..., k1, k2]  +  bar
          ↓ 合并
    [..., k1, k2']       ← k2 被 k2' 替换

  情况B：无包含
    [..., k1, k2]  +  bar
          ↓ 追加
    [..., k1, k2, bar']  ← bar 作为新 K 线加入
```

### 2.3 Step 3：更新笔（__update_bi）

这是最复杂的部分，是整个算法的核心。

```python
def __update_bi(self):
    if len(self.bars_ubi) < 3:
        return  # K 线不足，无法识别分型
    
    # ===== 第一笔的查找（bi_list 为空时） =====
    if not self.bi_list:
        self._find_first_bi()
        return
    
    # ===== 后续笔的查找 =====
    bi = check_bi(self.bars_ubi)
    if bi:
        self.bi_list.append(bi)
        # 更新 bars_ubi：截取 bi 之后的 K 线
        self.bars_ubi = bi.bars[-2:] + remaining_bars
    
    # ===== 笔破坏检测 =====
    self._check_bi_destruction()
```

#### 第一笔查找的特殊逻辑

```python
"""
第一笔查找流程：

1. 在 bars_ubi 中找到所有分型
2. 取第一个分型 fx_a
3. 如果有多个相同类型的分型在 fx_a 附近：
   - 底分型 → 找这些底分型中最低的
   - 顶分型 → 找这些顶分型中最高的
4. 从选定的 fx_a 位置截取 bars_ubi
5. 调用 check_bi 尝试识别第一笔

这个"找极值"的逻辑确保了第一笔的起点是真正的转折点，
而不是半山腰上的一个小波动。
"""
```

### 2.4 笔破坏-回退-重建机制

```python
def _check_bi_destruction(self):
    """检测最后一笔是否被后续走势破坏"""
    if not self.bi_list:
        return
    
    last_bi = self.bi_list[-1]
    
    # 判断是否被破坏
    destroyed = False
    if last_bi.direction == Direction.Up:
        # 向上笔：未完成 K 线出现更高的高点
        if self.bars_ubi[-1].high > last_bi.high:
            destroyed = True
    else:
        # 向下笔：未完成 K 线出现更低的低点
        if self.bars_ubi[-1].low < last_bi.low:
            destroyed = True
    
    if destroyed:
        # 回退：将 last_bi 的 K 线回退到 bars_ubi
        # 注意：保留 last_bi.bars 的前部分，与当前 bars_ubi 合并
        self.bars_ubi = last_bi.bars[:-2] + self.bars_ubi_from(last_bi.bars[-2].dt)
        self.bi_list.pop(-1)  # 删除被破坏的笔
```

#### 笔破坏的完整示例

```
初始状态：bi_list = [bi_1(向上), bi_2(向下)]
         bars_ubi = [k10, k11, k12]（3根 K 线，还在寻找新笔）

时间步 T：新来 K13

  bi_2 = 向下笔，从高点 G 到低点 D
  
  如果 K13.low < D.low：
    → bi_2 被破坏！
    
    【回退前】                    【回退后】
    bi_list: [..., bi_2]          bi_list: [...]
    bars_ubi: [k10,k11,k12,k13]  bars_ubi: [bi_2的K线 + k10~k13]
    
    回退后，算法重新在更长的 bars_ubi 上寻找笔
    可能找到一根更长的下降笔，其终点在 K13 以后

图示：
价格
  │    G ← bi_2 原始终点
  │   ╱ ╲
  │  ╱   ╲       K13
  │ ╱     ╲     ╱
  │╱       D   ╱  ← bi_2 原始终点（已被破坏）
  │         ╲ ╱
  │          ★ ← 新的更低点，bi_2 需要延伸到这里
```

> 💡 **核心哲学**：笔破坏机制体现了"走势终完美"的思想——一笔只有在真正完成后才会被最终确认。在完成之前，任何新的价格运动都可能改变已有的判断。

### 2.5 Step 4：数量控制

```python
    # Step 4: 内存管理
    if len(self.bi_list) > self.max_bi_num:
        # 只保留最近的 max_bi_num 根笔
        self.bi_list = self.bi_list[-self.max_bi_num:]
        # 同步裁剪 bars_raw：保留第一笔开始之后的 K 线
        first_bi_dt = self.bi_list[0].fx_a.elements[0].dt
        self.bars_raw = [b for b in self.bars_raw if b.dt >= first_bi_dt]
```

> **为什么需要数量控制？** 在实盘场景中，CZSC 对象可能运行数月甚至数年。不控制数量的话，内存会不断增长。默认保留最近 50 笔（约几百到几千根 K 线），足以覆盖当前分析需求。

## 三、关键派生属性

### 3.1 finished_bis vs bi_list

```python
@property
def finished_bis(self) -> List[BI]:
    """已完成的笔
    
    与 bi_list 的区别：
    - 当 bars_ubi 中有 ≥5 根 K 线时，bi_list[-1] 可能还不稳定
    - finished_bis 排除了这种不稳定的最后一笔
    
    使用场景：
    - 做交易决策时，用 finished_bis（更安全）
    - 做研究分析时，用 bi_list（更完整）
    """
    if len(self.bars_ubi) >= 5:
        return self.bi_list[:-1]
    return self.bi_list
```

```
为什么 bars_ubi ≥ 5 根时最后一笔不稳定？

bars_ubi 中 5 根 K 线意味着至少可以构成一个分型（3根）+部分走势，
这时候有可能触发笔破坏的条件。换句话说：

  bars_ubi < 5  → 不太可能产生新分型 → 最后一笔基本稳定
  bars_ubi ≥ 5  → 可能产生新分型       → 最后一笔有可能被推翻
```

## 四、多级别联立分析

### 4.1 BarGenerator：K 线合成引擎

`BarGenerator` 负责将低级别 K 线（如 1 分钟）实时合成为高级别 K 线（如 5 分钟、30 分钟、日线）：

```python
from czsc.core import BarGenerator

bg = BarGenerator(
    base_freq='1分钟',                         # 基础周期
    freqs=['5分钟', '15分钟', '30分钟', '日线'],  # 目标周期
    max_count=5000,                             # 每个周期最大K线数
    market='A股',                               # 市场类型（影响交易时间判断）
)
```

#### 合成算法

```python
"""
K 线合成流程（每个目标周期）：

1. 计算 freq_end_time(bar.dt, target_freq, market)
   → 得到当前 bar 属于哪根高级别 K 线

2. 如果 freq_end_time 与最后一根高级别 K 线时间不同：
   → 创建新的高级别 K 线
   → open/high/low/close/vol/amount 直接取 bar 的值

3. 如果 freq_end_time 与最后一根高级别 K 线时间相同：
   → 合并到当前高级别 K 线
   → open: 保持原值（第一笔交易的开盘）
   → close: 取 bar.close（最新收盘价）
   → high: max(已有.high, bar.high)
   → low: min(已有.low, bar.low)
   → vol: 累加
   → amount: 累加
"""
```

#### freq_end_time 的精确计算

```python
"""
分钟级别：通过"分钟切分表"(minutes_split.feather)精确查表
  
  例如 10:03 这一根 1 分钟 K 线：
  - 属于 5 分钟的 10:05 这根
  - 属于 15 分钟的 10:15 这根
  - 属于 30 分钟的 10:30 这根

日线及以上：
  - 周线 → 本周五
  - 月线 → 本月最后一天
  - 季线 → 本季最后一天
  - 年线 → 12月31日

市场类型影响：
  - A股：交易时间 9:30-11:30, 13:00-15:00
  - 期货：包含夜盘交易时段
  - 默认：不做交易时间过滤
"""
```

### 4.3 级别共振检测

```python
def detect_resonance(analyzers: dict, direction: str = "向上") -> dict:
    """
    多级别共振检测
    
    参数：
        analyzers: {freq_name: CZSC对象} 字典
        direction: 目标方向 "向上" 或 "向下"
    
    返回：
        {
            'is_resonance': bool,       # 是否形成共振
            'resonance_count': int,     # 共振级别数量
            'total_levels': int,        # 总级别数
            'details': dict,            # 各级别详情
        }
    """
    details = {}
    aligned = 0
    total = 0
    
    for freq_name, czsc_obj in analyzers.items():
        if not czsc_obj.bi_list:
            continue
        total += 1
        last_bi = czsc_obj.bi_list[-1]
        is_aligned = last_bi.direction.value == direction
        details[freq_name] = {
            'direction': last_bi.direction.value,
            'aligned': is_aligned,
            'power_price': last_bi.power_price,
        }
        if is_aligned:
            aligned += 1
    
    return {
        'is_resonance': aligned >= 3,  # 至少3个级别方向一致
        'resonance_count': aligned,
        'total_levels': total,
        'details': details,
    }

# 使用示例
result = detect_resonance(analyzers, "向上")
if result['is_resonance']:
    print(f"⚡ 多级别共振！{result['resonance_count']}/{result['total_levels']} 个级别向上")
```

```python
伪代码：
"""

def multi_level_decision(czsc_daily, czsc_30m, czsc_5m):

    # Step 1: 日线定方向
    daily_trend = classify_trend(czsc_daily)  # '上涨'/'下跌'/'盘整'

        return 'WAIT'  # 日线下跌，不做多
    
    # Step 2: 30分钟确认位置

        if last_fx.mark.value == '底分型' and last_fx.power_str == '强':

    return 'WAIT'
```

## 五、Rust/Python 混合架构

### 5.1 设计决策

```python
"""
为什么采用混合架构？

问题背景：Python 的性能在处理大量 K 线数据时存在瓶颈
  - 100 万根 K 线的分析可能需要数十秒
  - 多品种多周期的实时计算更是性能挑战

可选方案：
  A. 纯 Python → 开发快，但慢
  B. 纯 Rust → 性能好，但生态差，开发慢
  C. 混合架构 → 核心算法用 Rust，接口和策略用 Python

选择理由（C）：
  - 核心算法（包含关系处理、分型识别、笔识别）是 CPU 密集型 → Rust
  - 策略逻辑、信号计算需要灵活性 → Python  
  - 用户界面、可视化、数据源接口依赖 Python 生态 → Python

代价：
  - 开发者需要维护两套代码
  - 用户需要编译 Rust 库（但提供了预编译包）
```

## 八、边界情况与陷阱

### 8.1 问题诊断决策树

遇到"分析结果不对"时，按以下顺序排查：

```
分析结果不符合预期？
    │
    ├── 笔数量为 0
    │   ├── K 线数量 < 20？→ 数据不足，需要更多 K 线
    │   ├── 全部涨停/跌停？→ 正常（无分型可识别）
    │   └── 检查 id 是否升序、dt 是否排序
    │
    ├── 笔数量过多（很碎）
    │   ├── min_bi_len 太小？→ 增大到 7-8
    │   ├── 数据周期太小？→ 用更大级别分析
    │   └── 市场本身在剧烈震荡 → 正常
    │
    ├── 最后一笔反复变化
    │   ├── 正常！→ 这是笔破坏机制在工作
    │   ├── 用 finished_bis 代替 bi_list → 更稳定
    │   └── 等待更多 K 线确认
    │
    └── Rust 版本和 Python 版本结果不同
        ├── 检查 rs-czsc 版本是否最新
        ├── 提交 Issue 到 GitHub
        └── 临时方案：export CZSC_USE_PYTHON=1
```

### 8.2 数据质量问题

| 问题 | 症状 | 解决方案 |
|------|------|----------|
| K 线 id 不升序 | 分析结果混乱 | 确保 id 严格递增 |
| 时间不连续 | 出现异常长的笔 | 检查是否有缺失数据 |
| 价格为 0 或负数 | 程序异常 | 数据清洗时过滤 |
| 成交量为 0 | 分型力度计算异常 | 停牌等特殊情况，建议跳过 |
| 高点 < 低点 | 违反基本定义 | 数据源 bug，清洗修复 |

### 8.3 包含关系处理的边界

```python
"""
边界情况1: 连续多根K线相互包含的极端情况

  K1  K2  K3  K4  K5  K6   ← 全部互相包含
  ┌┐  ┌┐  ┌┐  ┌┐  ┌┐  ┌┐
  ││  ││  ││  ││  ││  ││
  ││  ││  ││  ││  ││  ││
  └┘  └┘  └┘  └┘  └┘  └┘

  → 最终合并为 1 根 NewBar
  → elements 字段包含所有 6 根原始 K 线
  → 如果超过 100 根被合并，elements 只保留最近 100 根

边界情况2: 第一根 K 线和第二根 K 线就有包含关系

  此时方向无法确定（至少需要两根无包含K线确定方向）
  解决：先按原始顺序加入，等第三根K线再回溯处理

边界情况3: 高点完全相等

  K1.high == K2.high → 不处理包含，保持原样
  原因：等高时方向不确定，强行合并可能引入错误
"""
```

### 8.4 笔识别的特殊场景

```python
"""
极端场景: 涨停/跌停连续板

  涨停板：每根K线 Open=Close=High=Low
  → 无法形成分型（没有凸起或凹陷）
  → 可能出现非常长时间没有笔的情况
  
  解决：这是正常的 —— 连续涨停确实没有"走势结构"可言

极端场景: 盘整时分型密集

  价格在狭窄区间内震荡
  → 产生大量分型和小笔
  → min_bi_len 帮助过滤，但可能仍然偏多
  
  建议：结合更大级别（日线/周线）来判断，小级别的细碎笔不做交易依据
"""
```

### 练习 3（创造型 ⭐⭐⭐⭐）：设计"智能 min_bi_len"

当前 `min_bi_len` 是全局固定值。设计一个自适应算法，根据近期市场波动自动调整：

要求：
1. 波动大时增大 `min_bi_len`（过滤更多噪音）
2. 波动小时减小 `min_bi_len`（捕获更多细节）
3. 给出明确的波动度量指标和阈值

<details>
<summary>参考思路</summary>

```python
import numpy as np

def adaptive_min_bi_len(bars, lookback=100):
    """自适应最小笔长度
    
    思路：用 ATR（真实波动幅度均值）衡量市场波动性
    
    ATR 高 → 市场剧烈波动 → 需要更长的笔来过滤噪音 → 增大 min_bi_len
    ATR 低 → 市场平静 → 可以用更短的笔捕获细节 → 减小 min_bi_len
    """
    if len(bars) < lookback:
        return 6  # 数据不足，用默认值
    
    recent = bars[-lookback:]
    
    # 计算 ATR
    true_ranges = []
    for i in range(1, len(recent)):
        tr = max(
            recent[i].high - recent[i].low,
            abs(recent[i].high - recent[i-1].close),
            abs(recent[i].low - recent[i-1].close)
        )
        true_ranges.append(tr)
    
    atr = np.mean(true_ranges)
    avg_price = np.mean([b.close for b in recent])
    relative_atr = atr / avg_price * 100  # 相对 ATR（百分比）
    
    # 映射到 min_bi_len
    if relative_atr > 3.0:    # 高波动
        return 8
    elif relative_atr > 1.5:  # 中波动
        return 6
    else:                     # 低波动
        return 5
```

## 十、核心公式速查

```
包含关系判断:
  include = (k2.H≥k3.H ∧ k2.L≤k3.L) ∨ (k2.H≤k3.H ∧ k2.L≥k3.L)

方向判定:
  dir = Up if k1.H < k2.H else Down

向上合并:
  H' = max(k2.H, k3.H),  L' = max(k2.L, k3.L)

向下合并:
  H' = min(k2.H, k3.H),  L' = min(k2.L, k3.L)

顶分型:
  k1.H < k2.H > k3.H  ∧  k1.L < k2.L > k3.L

底分型:
  k1.H > k2.H < k3.H  ∧  k1.L > k2.L < k3.L

成笔条件:
  len(bars) ≥ min_bi_len  ∧  ¬ab_include(fx_a, fx_b)

中枢参数:
  ZG = min(bi[0].H, bi[1].H, bi[2].H)
  ZD = max(bi[0].L, bi[1].L, bi[2].L)
  ZZ = (ZG + ZD) / 2
  valid = ZG ≥ ZD
```
